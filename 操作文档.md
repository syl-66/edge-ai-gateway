## 1. 项目概述

### 这个项目做了什么

```
你对着电脑说: "现在温度多少？灯光太暗了"

Agent 理解你的话 → 通过 MQTT 问开发板 → 开发板读传感器 → 
数据返回 → Agent 告诉你 "26.5°C" → Agent 自动开灯
```

| **SQLite** | 建表/INSERT/查询统计/数据清理/WAL 模式 |
| **MQTT** | libmosquitto 发布/订阅/QoS1/retained/Will Message |

## 4. 第1步：LED 亮起来（GPIO sysfs）

> **对应教程**：第 2 周 — 文件 I/O 与设备操作

### 4.1 原理

GPIO (General Purpose Input/Output) 是最简单的硬件接口。Linux 通过 sysfs 把每个 GPIO 引脚暴露为 `/sys/class/gpio/` 目录下的文件：

```
echo 2 > /sys/class/gpio/export          # 告诉内核"我要用 GPIO2"
→ /sys/class/gpio/gpio2/ 目录出现
  ├── direction   (写 "in" 或 "out")
  └── value       (读=获取电平, 写="1"/"0"=设置输出)
```

### 4.2 代码

```c
// 打开 /sys/class/gpio/gpio2/value → 写 "1" → LED 亮
int fd = open("/sys/class/gpio/gpio2/value", O_WRONLY);
write(fd, "1", 1);   // 高电平 → LED 亮
close(fd);
```

**完整封装**见 `actuator/device_manager.c` 的 `gpio_write_val()` 函数。

### 4.3 为什么不用 fopen

`/sys/class/gpio/gpio2/value` 不是普通文件，它是**内核驱动的接口**。用 `fopen` 会导致用户态缓冲——你 `fprintf(fp, "1")` 之后数据可能还在缓冲区没发到内核，LED 没反应。必须用系统调用 `open/write` 直接进内核。

## 5. 第2步：读 DHT11 温湿度（GPIO bit-banging）

> **对应教程**：第 2 周 — 文件 I/O + 第 2.5 周 — 1-Wire 协议

### 5.1 DHT11 是什么

DHT11 是单总线（1-Wire）温湿度传感器，只有一根数据线。通信完全靠主机拉低/拉高这根线来控制时序，传感器在特定时间窗口内返回数据。

### 5.2 通信时序

```
1. 主机发起: 拉低 18ms → 拉高 20-40us → 释放总线
2. DHT11 响应: 拉低 80us → 拉高 80us
3. DHT11 发送 40 bit 数据:
   - 每个 bit 以 50us 低电平开始
   - 高电平 26-28us = 0, 70us = 1
4. 40bit = 湿度整数(8) + 湿度小数(8) + 温度整数(8) + 温度小数(8) + 校验(8)
```

### 5.3 代码

```c
// 见 sensor/sensor_manager.c 的 dht11_read() 函数
// 核心逻辑:
// 1. open("/sys/class/gpio/gpio17/value")  → 操作 GPIO 电平
// 2. usleep() 控制时序 → 18ms 拉低、30us 拉高等
// 3. 逐 bit 读取 40 位数据
// 4. 校验和验证
// 5. 计算物理值
```

### 5.4 注意

用户态用 `usleep()` 做 us 级延时**精度不够**（Linux 调度粒度约 10ms），高可靠性场景应该用内核 DHT11 驱动。我们这个项目是学习协议原理，面试能讲清时序就行。

## 6. 第3步：读 BH1750 光照传感器（I2C）

> **对应教程**：第 2.5 周 — I2C 协议

### 6.1 I2C 是什么

I2C (Inter-Integrated Circuit) 是两线制同步串行总线：
- **SCL**：时钟线（主机产生）
- **SDA**：数据线（双向，需要上拉电阻）

每个从设备有一个 7-bit 地址。主机通过地址选择要通信的从机。

### 6.2 为什么用户态能直接操作 I2C

Linux 内核提供了 `i2c-dev` 驱动，允许用户态程序通过 `/dev/i2c-1` 设备文件直接收发 I2C 数据，**不需要为每个传感器写内核驱动**。

```
你的程序
  ↓ write(fd, &cmd, 1)
i2c-dev.c (内核)
  ↓ i2c_master_send()
i2c-imx.c (I2C 控制器驱动)
  ↓ 操作寄存器发 SCL/SDA 波形
BH1750 传感器
```

### 6.3 代码

```c
// === 1. 打开 I2C 总线 ===
int fd = open("/dev/i2c-1", O_RDWR);

// === 2. 告诉内核"我要和地址 0x23 的设备通信" ===
ioctl(fd, I2C_SLAVE, 0x23);
// 之后所有 write/read 都针对这个从机

// === 3. 发测量指令 (1 字节) ===
uint8_t cmd = 0x10;  // BH1750 高分辨率模式
write(fd, &cmd, 1);
// 内核: 发送 START → 0x23(写) → 0x10 → STOP

// === 4. 等 180ms (传感器测量需要时间) ===
usleep(180000);

// === 5. 读 2 字节结果 ===
uint8_t buf[2];
read(fd, buf, 2);
// 内核: 发送 START → 0x23(读) → 读 2 字节 → STOP

// === 6. 计算光照值 ===
uint16_t raw = (buf[0] << 8) | buf[1];
double lux = raw / 1.2;
```

### 6.4 面试重点

> "为什么用 `ioctl(I2C_SLAVE)`？因为这个 ioctl 告诉内核 i2c-dev 驱动之后所有 `read/write` 针对哪个从机地址。内核会自动处理 START/STOP 信号和地址字节，你不需要手动构造 I2C 帧。"

## 7. 第4步：读 PMS5003 PM2.5 传感器（UART）

> **对应教程**：第 2.5 周 — UART 协议

### 7.1 原理

PMS5003 是激光粉尘传感器，通过 UART 主动上报数据。你不需要发送查询指令，它每隔 200-800ms 自动发一帧 32 字节的数据。

### 7.2 串口配置

```c
// === 1. 打开串口 ===
int fd = open("/dev/ttyS2", O_RDWR | O_NOCTTY);  // O_NOCTTY 必须加

// === 2. 获取当前配置 ===
struct termios opt;
tcgetattr(fd, &opt);

// === 3. 设波特率 9600 ===
cfsetispeed(&opt, B9600);
cfsetospeed(&opt, B9600);

// === 4. 设 8N1 (8数据位, 无校验, 1停止位) ===
opt.c_cflag &= ~PARENB;   // 无校验
opt.c_cflag &= ~CSTOPB;   // 1 停止位
opt.c_cflag &= ~CSIZE;    // 清除数据位
opt.c_cflag |= CS8;       // 8 数据位

// === 5. 设为原始模式 (不做任何字符处理) ===
opt.c_lflag &= ~(ICANON | ECHO | ECHOE | ISIG);   // 非规范模式
opt.c_iflag &= ~(IXON | IXOFF | IXANY);            // 无软件流控
opt.c_oflag &= ~OPOST;                              // 原始输出

// === 6. 设超时 ===
opt.c_cc[VMIN] = 0;     // 最少读 0 字节就返回
opt.c_cc[VTIME] = 10;   // 最多等 1.0 秒

// === 7. 应用配置 ===
tcsetattr(fd, TCSANOW, &opt);
```

### 7.3 数据帧解析

```
PMS5003 数据帧 (32 字节):
Byte 0:   0x42 (起始符1, 固定)
Byte 1:   0x4D (起始符2, 固定)
Byte 2-3: 帧长度 (大端, 固定 0x001C = 28)
Byte 4-5: PM1.0 浓度
Byte 6-7: PM2.5 浓度  ★ 重点关注: (Byte[6]<<8) | Byte[7]
Byte 8-9: PM10  浓度  ★ 重点关注: (Byte[8]<<8) | Byte[9]
...
Byte 30-31: 校验和 (前 30 字节累加取低 16 位)
```

### 7.4 接收策略

PMS5003 主动上报，我们用 **epoll** 带超时接收，找到帧头 `0x42 0x4D` 后读 32 字节，验证校验和，提取 PM2.5 和 PM10。

## 9. 第6步：发射红外信号（NEC 协议）

> **对应教程**：第 2 周 — GPIO 输出

### 9.1 NEC 协议简介

NEC 是家用红外遥控最常用的编码协议：

```
1. 引导码: 9ms 载波 (38kHz) + 4.5ms 空闲
2. 数据: 地址(8bit) + 地址反码(8bit) + 命令(8bit) + 命令反码(8bit)
3. 逻辑 0: 560us 载波 + 560us 空闲
4. 逻辑 1: 560us 载波 + 1690us 空闲
5. 结束位: 560us 载波
```

### 9.2 代码

`actuator/device_manager.c` 的 `device_ir_send()`：
- 解析 NEC 十六进制编码 ("0x00FFA25D")
- `ir_carrier_us()` 函数：用 GPIO 模拟 38kHz 载波（周期 26us，占空比 1/3）
- 按 NEC 时序发送引导码 + 32bit 数据 + 结束位

## 10. 第7步：传感器数据存 SQLite

> **对应教程**：第 4 周 — 线程同步（mutex 保护数据库）

### 10.1 为什么存本地

- MQTT 上报可能丢（网络抖动），本地 SQLite 作为可靠备份
- Agent 能问"过去 24 小时平均温度是多少"，只靠 MQTT 做不到
- 设备断网时数据不丢，等网络恢复后可以补传

### 10.2 WAL 模式

```sql
PRAGMA journal_mode=WAL;
```

Write-Ahead Logging：写入操作先写 WAL 文件，读取操作仍然读主数据库。**采集线程在写的同时，Agent 查询线程可以读**，读写不互斥。这是嵌入式 SQLite 最重要的性能优化。

### 10.3 代码核心

```c
// 建表
CREATE TABLE IF NOT EXISTS sensor_data (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp   INTEGER NOT NULL,
    temperature REAL,
    humidity    REAL,
    illuminance REAL,
    pm25        INTEGER,
    pm10        INTEGER,
    event       TEXT DEFAULT 'auto'
);
CREATE INDEX IF NOT EXISTS idx_timestamp ON sensor_data(timestamp);

// 写入 (用 ? 占位符防 SQL 注入)
INSERT INTO sensor_data (timestamp, temperature, ...) VALUES (?, ?, ...);

// 统计 (面试重点)
SELECT AVG(temperature), MIN(temperature), MAX(temperature)
FROM sensor_data WHERE timestamp >= ?;

// 清理 (防止磁盘写满)
DELETE FROM sensor_data WHERE timestamp < ?;
```

### 10.4 线程安全

多线程同时写 SQLite 需要保护。我们用一个全局 `pthread_mutex_t` 锁住所有 SQLite 操作，保证同一时刻只有一个线程在操作数据库。

## 11. 第8步：MQTT 发布传感器数据

> **对应教程**：第 6 周 — MQTT 协议

### 11.1 MQTT 选型理由

为什么不用 HTTP？
- HTTP 每次请求需要 TCP 三次握手，开销大
- MQTT 一条 TCP 连接复用所有消息，心跳只有 2 字节
- MQTT 有 QoS 机制保证消息送达
- MQTT 的发布/订阅模型天然解耦传感器和 Agent

### 11.2 消息格式

我们设计了一个简单的 JSON-RPC 风格协议：

```json
// 传感器上报
{
  "id": "evt_001",
  "method": "sensor.report",
  "params": {
    "timestamp": "2026-07-01T12:00:00",
    "device": "imx6ull-gateway-001",
    "sensors": [
      {"name": "temperature", "value": 26.5, "unit": "celsius"},
      {"name": "humidity",    "value": 45.2, "unit": "percent"},
      {"name": "illuminance", "value": 500.0, "unit": "lux"},
      {"name": "pm25",        "value": 35, "unit": "ug/m3"}
    ]
  }
}
```

### 11.3 Topic 设计

```
edge/imx6ull-gateway-001/sensor/report   ← 传感器数据上报
edge/imx6ull-gateway-001/tool/call       ← Agent 下发工具调用
edge/imx6ull-gateway-001/tool/result     ← 设备回传执行结果
edge/imx6ull-gateway-001/heartbeat       ← 心跳 (设备在线状态)
```

- `mqtt_publish()` — 发消息
- `mqtt_publish_tool_registry()` — 发布 retained 消息告诉 Agent "我有哪些工具"

## 13. 第10步：多线程架构组装

> **对应教程**：第 4 周 — 多线程

### 13.1 为什么用多线程

如果单线程轮询：
```
读传感器(2s) → 发 MQTT → 等接收 → 读传感器... 
每次等待期间 CPU 空转
```

多线程分离：
```
Thread 1: 传感器采集    每 2s 跑一次          (周期性强)
Thread 2: MQTT 接收     阻塞等消息            (事件驱动)
Thread 3: MQTT 发送     消费消息队列           (有数据就发)
Thread 4: 心跳          每 30s 发一次         (周期性强)
主线程:   信号处理+监控  wait + 清理          (调度)
```

### 13.2 线程间通信

用 **POSIX 消息队列** (`mq_open/mq_send/mq_receive`)：

```
采集线程 → mq_send → [sensor_mq] → mq_receive → 发送线程 → MQTT
Agent指令 → mqtt接收线程 → mq_send → [tool_mq] → mq_receive → 发送线程 → MQTT
```

选择 mq 而不是全局变量 + mutex 的原因：
- mq 自带同步，不需要额外的锁
- mq 有消息边界，每条消息独立完整
- mq 支持超时接收（`mq_timedreceive`）

### 13.3 线程安全退出

```c
static volatile sig_atomic_t g_running = 1;

void sig_handler(int sig) { g_running = 0; }

// 每个线程循环: while(g_running) { ... }
// 主线程收到 SIGINT → g_running=0 → 所有线程依次退出
// 主线程 pthread_join 等所有线程 → 清理资源 → 退出
```

## 17. 常见问题排查

### 17.1 I2C 检测不到设备

```bash
# 1. 确认 I2C 总线已启用
ls /dev/i2c-*

# 2. 确认接线 (SDA/SCL 有没有接反)
# 3. 确认上拉电阻 (I2C 必须 4.7kΩ 上拉到 3.3V)
# 4. 扫描 I2C 总线
sudo i2cdetect -y 1

# 5. 如果扫描不到，用示波器看 SDA/SCL 波形
```

### 17.2 MQTT 连不上

```bash
# 1. 确认 Broker 在运行
ps aux | grep mosquitto

# 2. 确认端口开放
nc -zv localhost 1883

# 3. 看 Broker 日志
tail -f /var/log/mosquitto/mosquitto.log

# 4. 用命令行工具测试
mosquitto_sub -t "test" -v &
mosquitto_pub -t "test" -m "hello"
```

### 17.3 串口没数据

```bash
# 1. 确认串口设备存在
ls /dev/ttyS*

# 2. 确认波特率匹配 (PMS5003 默认 9600)
stty -F /dev/ttyS2 9600

# 3. 用 cat 看原始数据
cat /dev/ttyS2 | xxd | head -10
# 应该看到 42 4D 开头的帧

# 4. 确认 GPIO 串口复用 (i.MX6ULL 的 pin 可以复用为 GPIO/UART/I2C 等)
# 检查设备树中 UART3 是否正确配置
```

### 17.4 GPIO 操作 Permission Denied

```bash
# 这是因为普通用户没有 GPIO sysfs 的操作权限
# 方法1: sudo
sudo ./edge_gateway

# 方法2: 用户加入 gpio 组
sudo usermod -aG gpio $USER
# 重新登录生效
```

### 17.5 交叉编译的程序在开发板上不能运行

```bash
# 1. 确认是 ARM 架构
file edge_gateway
# 输出: ELF 32-bit LSB executable, ARM, ...

# 2. 确认动态库都存在
# 在 PC 上:
arm-linux-gnueabihf-ldd edge_gateway
# 确认 .so 文件在开发板对应位置

# 3. 确认权限
chmod +x edge_gateway
```
